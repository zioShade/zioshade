# Changelog

All notable changes to glslpp are documented here. The format is loosely based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and the project adheres to [SemVer](https://semver.org/) on the public API exported from `src/root.zig`.

## [Unreleased]

### Fixed (silent-wrong elimination, all oracle-gated)
- **MSL: `packHalf2x16`/`unpackHalf2x16` emitted non-existent builtins → non-compiling Metal (#gaps packHalf; spirv-cross-gated).** The MSL backend mapped GLSL.std.450 `PackHalf2x16`(58)→`pack_float_to_half2x16` and `UnpackHalf2x16`(62)→`unpack_half2x16_to_float` — but **those MSL functions do not exist**: Metal's `pack_float_to_*`/`unpack_*_to_float` cover only the unorm/snorm formats, and half conversion is done through `half` + `as_type`. So `packHalf2x16`/`unpackHalf2x16` produced MSL that does not compile (silent-wrong: glslpp exits 0). The unorm/snorm variants (`packUnorm2x16`/`packSnorm2x16`/`packUnorm4x8`/`packSnorm4x8` and their unpacks) use real MSL builtins and were already correct. **Fix (`src/spirv_to_msl.zig`):** the `.ExtInst` arm now emits the spirv-cross idiom — `packHalf2x16(vec2)` → `as_type<uint>(half2(x))` and `unpackHalf2x16(uint)` → `float2(as_type<half2>(x))` (GLSL fixes these types, so the result/operand types are hardcoded); `half2(vec2)` performs the f32→f16 narrowing and `as_type` the bit-reinterpret, matching GLSL semantics (NaN/inf preserved). The dead `std450ToMsl` 58/62 mappings were removed so a future bypass honest-errors rather than emitting the phantom builtin. New regression tests (`tests/msl_tests.zig` T-pack.1 asserts the half idiom + `assertNotContains` the invented names; T-pack.2 guards that the real unorm/snorm builtins are undisturbed), verified RED→GREEN. Adversarial review (bdd-reviewer, APPROVE): confirmed the `as_type` reinterpret + `half2` narrowing semantics vs the oracle and the safe dead-mapping removal; hardened by hardcoding the declared LHS types (`uint`/`float2`) and tightening the assertions. Verified MSL 158 / GLSL 139 / HLSL 802 / WGSL 156; conformance **PASS 2087 / FAIL 0 / XFAIL 13** (MSL-text-only change).
- **MSL: `findMSB`/`findLSB` lowered to raw `clz`/`ctz` — returned the wrong value (#gaps findMSB/findLSB; spirv-cross-gated).** The MSL backend mapped GLSL.std.450 `FindILsb`(73)→`ctz`, `FindSMsb`(74)→`clz`, `FindUMsb`(75)→`clz` (with a literal "simplified; may need adjustment" comment) — silent-wrong: `findMSB` must return the most-significant-bit **index** (`31 - clz` for 32-bit), not the leading-zero **count**, so `findMSB(1u)` is `0` but `clz(1u)` is `31`; `findLSB(0)` must be `-1` but `ctz(0)` is `32`; and signed `findMSB` must flip negatives first (most-significant 0 bit). HLSL (`firstbithigh`/`firstbitlow`) and WGSL (`firstLeadingBit`/`firstTrailingBit`) were already correct; only MSL was wrong. **Fix (`src/spirv_to_msl.zig`):** the `.ExtInst` arm now inlines the exact spirv-cross helper math — `findLSB` → `select(ctz(x), -1, x==0)`; `findMSB(unsigned)` → `select(clz(T(0)) - (clz(x)+1), -1, x==0)` (`clz(T(0))` is the bit width, so this is the MSB index for any width); `findMSB(signed)` → flip negatives (`v = x<0 ? ~x : x`) then the same index. The math is computed in the **argument** type and cast to the result type (glslang gives the unsigned-operand forms a signed result, so this avoids MSL-rejected mixed int/uint arithmetic), mirroring spirv-cross's `int(spvFindUMSB(x))`; `select`/`clz`/`ctz` are componentwise so scalars and vectors share one path. The dead `std450ToMsl` 73/74/75 mappings were removed so a future bypass honest-errors rather than silently emitting the wrong count. New regression tests (`tests/msl_tests.zig` T-bits.0/.1/.2/.3/.4: scalar findLSB/findMSB for int & uint, the signed negative-flip, and the componentwise vector path), each verified RED→GREEN; the pre-existing T11.2 test (which rubber-stamped the bare-`ctz`/`clz` output) was repurposed to assert the guarded idiom. Adversarial review (bdd-reviewer, REQUEST-CHANGES→resolved): independently verified the index math + signed flip + `select` argument order against the oracle for every sign/zero case; fixes applied = a defensive `words.len < 6` bound and the vector test (T-bits.4). Verified MSL 156 / GLSL 139 / HLSL 802 / WGSL 156; conformance **PASS 2087 / FAIL 0 / XFAIL 13** (MSL-text-only change).
- **MSL: EVERY atomic op emitted non-compiling Metal — the object was passed bare (no `(device|threadgroup atomic_T*)&` cast) and the SSA result temp was undeclared (#265; spirv-cross-gated; closes the #264 review follow-up).** Metal's `atomic_*_explicit` builtins require an address-space-qualified `atomic_T*`, but the MSL backend passed the bare struct member (`atomic_fetch_add_explicit(b.total, 37u, …)` — `b.total` is a plain `uint`, not an atomic pointer) and emitted the result as a bare assignment to an undeclared SSA temp (`v8 = atomic_…(…)`). Both make the output fail to compile; this affected *all* of `atomicAdd`/`Sub`/`And`/`Or`/`Xor`/`Min`/`Max`, `atomicExchange`, `atomicCompSwap`, and `AtomicFAddEXT`. **Fix (`src/spirv_to_msl.zig`):** new helpers `mslAtomicAddrSpace` (derives `device`/`threadgroup` from the pointer's SPIR-V storage class — Workgroup→threadgroup, StorageBuffer/Uniform→device), `mslAtomicTypeName` (`uint`→`atomic_uint`, `int`→`atomic_int`, `float`→`atomic_float`), and `mslAtomicObject` (returns the cast object `(<addrspace> <atomicT>*)&<member>`); every atomic arm now emits `{scalar} {rn} = atomic_…_explicit({cast-object}, {val}, memory_order_relaxed);` — both casting the object *and* declaring the result temp — byte-faithful to `spirv-cross --msl` (`uint _19 = atomic_fetch_add_explicit((device atomic_uint*)&b.total, 37u, memory_order_relaxed);`). The CompareExchange arm keeps its #263 spirv-cross do-while idiom but now uses the cast object and declares its result. The old `classifyMslAtomicPtr`/`MslAtomicPtr` union were removed. New regression tests (`tests/msl_tests.zig` T-atomic.5/.6/.7): device-SSBO `atomic_uint` cast + result declaration, threadgroup (`shared`) cast, and signed `atomic_int` cast — each verified RED→GREEN; the pre-existing T-atomic.1–.4 still hold. Adversarial review (bdd-reviewer, REQUEST-CHANGES→resolved): verified the storage-class decode (`OpTypePointer.words[2]`) and result-temp safety against the spirv-cross oracle; fixes applied = a stale comment referencing the removed helper, the `spirv.StorageClass.Workgroup` enum (was a bare `4`), and a Gap-B assertion added to the threadgroup test. **Known out of scope (documented, follow-up candidates):** image atomics still use the `&img[coord]` form (Metal needs `img.atomic_fetch_*(coord,…)` method syntax — a separate texture-atomic rework), and `atomic_float` (the `AtomicFAddEXT` path) requires Metal 3.0 with no `--metal-version` gating yet (float atomics were already non-compiling pre-fix, so no regression). Verified MSL 152 / GLSL 139 / HLSL 802 / WGSL 156; conformance **PASS 2087 / FAIL 0 / XFAIL 13** (MSL-text-only change).
- **MSL: `atomicCompSwap` emitted non-compiling / silently-wrong `atomic_compare_exchange_weak_explicit` — `&<literal>` (invalid lvalue) and the bool return assigned as the original value (#263; spirv-cross-gated; closes the #260 review follow-up).** Metal's `atomic_compare_exchange_weak_explicit(object, thread T* expected, T desired, …)` does **not** match GLSL `atomicCompSwap(mem, compare, data)`: `expected` is an in/out pointer needing an addressable mutable lvalue, and the builtin **returns a `bool` (success), not the original loaded value**. The MSL backend (`src/spirv_to_msl.zig`) emitted `v = atomic_compare_exchange_weak_explicit(ptr, &<compare>, <data>, …)` — so for a constant comparator it produced `&7u` (address-of a literal → does not compile in Metal), and it assigned the success-bool to the result variable even though `OpAtomicCompareExchange`'s result is the original value (silent-wrong when the result is used). **Fix:** emit the spirv-cross idiom — materialize a mutable local seeded with the comparator, retry on spurious `_weak` failure (the only CAS form Metal offers), and take the original value from the local: `T _cas_expected_N; do { _cas_expected_N = <compare>; } while (!atomic_compare_exchange_weak_explicit(ptr, &_cas_expected_N, <data>, …) && _cas_expected_N == <compare>); rn = _cas_expected_N;`. Applied to both the SSBO and image variants; byte-faithful to `spirv-cross --msl`. The sibling `atomicAdd`/`atomicExchange`/`AtomicFAddEXT` arms are untouched. New regression tests in `tests/msl_tests.zig` (T-atomic.1/.3 reworked to assert the local-lvalue form + retry; new T-atomic.4 covers a runtime non-constant comparator), each verified RED→GREEN. Adversarial review (APPROVE) verified the retry-loop CAS semantics and format-string argument tuples against the spirv-cross oracle, and confirmed two pre-existing systemic MSL-atomic gaps remain out of scope (the atomic object lacks spirv-cross's `(device atomic_uint*)&` cast, and the `v<N>` SSA result temp is undeclared — both affect *every* MSL atomic arm equally; filed as a follow-up). Verified MSL 149 / GLSL 139 / HLSL 802 / WGSL 156; conformance **PASS 2087 / FAIL 0** (MSL-text-only change).
- **GLSL & MSL: `atomicCompSwap` / `atomic_compare_exchange_weak` read the compare and data operands from the wrong SPIR-V words — silent-wrong compare-and-swap (#260; cross-backend sibling of the WGSL #170 fix).** `OpAtomicCompareExchange` has **two** memory-semantics operands (Equal + Unequal), so its layout is `[ptr][scope][eq-sem][uneq-sem][value(new/data)][comparator(compare)]` — data at `words[7]`, compare at `words[8]`. Both the GLSL (`src/spirv_to_glsl.zig`) and MSL (`src/spirv_to_msl.zig`) emitters read the value from `words[6]` (the Unequal-semantics constant) and the compare from `words[7]` (the actual data), so `atomicCompSwap(b.lock, 7u, 9u)` emitted GLSL `atomicCompSwap(b.lock, 9u, 64u)` and MSL `atomic_compare_exchange_weak_explicit(b.lock, &9u, 64u, …)` — comparing against the *data* (`9`) and storing a *memory-semantics constant* (`64`). Output is syntactically valid, so it was invisible to conformance (which validates SPIR-V, not backend text). HLSL already read the correct words (`[7]`/`[8]`) and was the reference. **Fix:** read data from `words[7]` and compare from `words[8]` in both backends (matching HLSL and the WGSL #170 fix); the SSBO and image (`imageAtomicCompSwap` / image `atomic_compare_exchange_weak_explicit`) variants share the decode and are now both correct. The sibling `atomicAdd`/`atomicExchange` ops (which correctly read `words[6]`) are untouched. New regression tests in `tests/glsl_tests.zig` and `tests/msl_tests.zig` (SSBO + image, distinctive non-colliding values: compare 7/data 9 and compare 5/data 3 vs the semantics constant 64) — each verified RED→GREEN. Also corrected a misleading `OpAtomicCompareExchange` operand comment in `src/codegen.zig`. Adversarial review (APPROVE) verified the operand indices against the SPIR-V spec independently, confirmed correct argument-slot mapping in all backends, and surfaced the image-variant text-coverage gap (closed by the new tests). Verified GLSL 139 / MSL 148 / HLSL 802 / WGSL 156; conformance **PASS 2087 / FAIL 0**.
- **WGSL: atomic ops emitted the wrong operand as their value/comparator — every `atomicAdd`/`Sub`/`And`/`Or`/`Xor`/`Min`/`Max`, `atomicExchange`, and `atomicCompSwap` was silently wrong unless the value happened to equal the memory scope (#170; naga-gated).** The hand-written WGSL atomic emitters read their value operand from `words[4]` — but that is the memory **scope**; the value sits at `words[6]` (after the scope and semantics operands). So `atomicAdd(b.total, 37u)` emitted `atomicAdd(&b.total, 1u)` (the Device-scope constant `1`), and the whole add/sub/and/or/xor/min/max family was wrong for any operand ≠ the scope; `atomicExchange(x, 42u)` emitted `1u`; `atomicCompSwap(p, 7u, 9u)` emitted `atomicCompareExchangeWeak(&p, 64u, 9u)` — using a memory-semantics constant (`64`) as the compare value. naga accepts all of these (valid syntax, wrong values), so it was invisible to conformance (which tests SPIR-V) and to the WGSL-acceptance corpus survey. **Fix (`src/spirv_to_wgsl.zig`):** read the value from `words[6]` for the RMW + exchange ops; for `OpAtomicCompareExchange` (which has **two** memory-semantics operands) read the new value from `words[7]` and the comparator from `words[8]`. Also bind the RMW result with `let` (was `var`), matching the other atomic emitters. New naga-validated regression tests (`tests/wgsl_tests.zig`) with deliberately non-scope values: `atomicAdd(…, 37u)`, `atomicExchange(…, 42u)`, `atomicCompSwap(…, 7u, 9u)` — each verified RED→GREEN. Adversarial review (APPROVE) confirmed no other WGSL atomic path is affected (`atomicLoad`/`atomicStore` go through the `OpLoad`/`OpStore` lowering, which is correct) and the `var`→`let` cleanup. Verified WGSL 156 / opt 29 / GLSL 134 / HLSL 802 / MSL 145 / correctness 32; conformance **PASS 2087 / FAIL 0**. **Filed follow-up #260:** GLSL + MSL share the (narrower) `atomicCompSwap` operand bug; HLSL is already correct.
- **WGSL: integer division/modulo by a literal zero emitted a type-invalid `let v: i32 = 0.0;` (#258; naga-gated; closes the #254 follow-up).** A pre-existing `emitBinOp` band-aid emitted `let NAME: T = 0.0;` for any division whose divisor *rendered* as a literal zero string. Integer constants render as bare `"0"`, so an integer divide-by-zero hit it → `let v: i32 = 0.0;`, which naga rejects ("expected `i32`, got `AbstractFloat`") — silent-wrong (glslpp exits 0, output doesn't parse). The unsigned (`4u / 0u`) and modulo (`4 % 0`) forms slipped past the band-aid entirely and were likewise naga-invalid. **Fix:** the string-based band-aid (and its dead `isLiteralZero` helper) is removed and replaced by a structural guard on the SPIR-V operands — an integer `/`/`%` whose **divisor is a constant zero** and **dividend is a constant** is genuinely unrepresentable (naga const-evaluates and rejects, and WGSL has no integer inf/nan) → **honest-error** (`error.UnsupportedOp`), matching the #252/#254 "unrepresentable → fail loud" policy. A **runtime** integer divide-by-zero (`b / 0`) is well-defined in WGSL (`x / 0 == x`) and now emits the plain division naga accepts, instead of the type-wrong `0.0`. New naga-validated regression tests (`tests/wgsl_tests.zig`): runtime `b / 0` validates with no `i32 = 0.0`; const `4 / 0`, `4u / 0u`, `4 % 0` each honest-error — verified RED→GREEN. Adversarial review (APPROVE, 0 crit/major): added an `inst.words.len >= 5` bound check and a 64-bit-constant limitation note. Verified WGSL 153 / opt 29 / GLSL 134 / HLSL 802 / MSL 145 / correctness 32; conformance **PASS 2087 / FAIL 0**. (Known pre-existing gap, out of scope: a *vector*-integer const division by a zero-vector — divisor is `OpConstantComposite`, not caught here.)
- **WGSL: unfolded constant division/mod by zero emitted as a literal `1.0f / 0.0f` / `1.0f % 0.0f` → naga const-evaluates and rejects it (#254; naga-gated; closes the #252 follow-up).** The frontend deliberately does not fold a constant division by zero, so `1.0/0.0` (and `mod(1.0,0.0)`) reached the WGSL backend as an `OpFDiv`/`OpFMod` of two `OpConstant`s and was emitted verbatim — naga then const-evaluates the literal expression and rejects it ("Float literal is infinite" / "is NaN"), silent-wrong (glslpp exits 0, output doesn't parse). **Fix:** `emitBinOp` in `src/spirv_to_wgsl.zig` now const-folds a scalar-float `+`/`-`/`*`/`/`/`%` of two 32-bit float constants whose IEEE result is non-finite into `bitcast<f32>(0x..u)` (a runtime value naga accepts), reusing #252's bit-pattern idiom; finite results, integer ops, vector ops, and f16 are untouched (the fold gates on a 32-bit-float `OpConstant` result). `emitBinOp` only emits function-body `let` statements (never a const-expression), so the bitcast is always valid here. New naga-validated regression tests (`tests/wgsl_tests.zig`): `1.0/0.0`→+inf bitcast, `0.0/0.0`→NaN bitcast, `mod(1.0,0.0)`→NaN bitcast (the `%`/FMod case caught by the adversarial review of the division fix), each verified RED before the fix. Verified WGSL 149 / opt 29 / GLSL 134 / HLSL 802 / MSL 145 / correctness 32; conformance **PASS 2087 / FAIL 0**.
- **Frontend: load-value cache sibling-aliasing residuals — stale shared/SSBO reads across a store/atomic/`++`/swizzle-assign (#248; closes the #235 follow-up residuals).** The load-value cache (`load_cache`/`global_load_cache` in `src/semantic.zig`) reused a cached load across a memory mutation it should have been invalidated by — silent-wrong on every backend (a stale value, glslpp exits 0). #235 invalidated SIBLING access chains into the same base, but keyed on the IMMEDIATE base via the per-BLOCK `ac_result_to_base` map, leaving four residuals: **(A)** a sibling load cached in a DOMINATING block (entry/loop-header → `global_load_cache`) was absent from the child block's map; **(B)** a MULTI-LEVEL nested lvalue (`b.arr[i].x` vs `b.arr[j].x`) resolved to different intermediate bases though the same root; **(C)** the MUTATED chain itself built in a dominating block (reused via `global_access_chain_cache`) was absent from the child block's map, so the gate skipped invalidation entirely; **(D)** the multi-component swizzle compound-assign (`v.xy *= k`) store-back was a fifth mutation site that never invalidated siblings at all. **Fix:** a new per-FUNCTION-persistent `chain_root_base` map records every access chain's transitive ROOT variable (resolved once at the single `emitAccessChain` chokepoint). All five mutation sites now call `invalidateAliasingChains(mutated_ptr)`, which resolves the root and drops every cached load through ANY chain into that root (plus the root's own load) — reaching dominating-block and nested siblings. The old write-only `ac_result_to_base` map was removed (superseded). Over-invalidation only forces a re-load (the optimizer re-CSEs immutable loads), never reuses a stale value. New frontend regression tests (`tests/optimizer_tests.zig`, opt + no-opt for each of A/B/C/D), each verified RED before the fix and GREEN after. Two adversarial review rounds: the first found residual C, the second found residual D — both folded in. Verified opt 29 / GLSL 134 / HLSL 802 / MSL 145 / WGSL 146 / correctness 32; conformance **PASS 2087 / FAIL 0**.
- **WGSL: non-finite float CONSTANTS emitted as the bare `inf`/`-inf`/`nan` identifier → naga-invalid output (#252; naga-gated).** WGSL has no inf/nan float literal, so a non-finite 32-bit float constant (an overflowing `1e40` → +inf, a folded `1e40 - 1e40` → NaN, etc.) was emitted as the bare identifier `inf`/`nan`, which naga rejects ("no definition in scope for identifier: `inf`") — silent-wrong (glslpp exits 0, output doesn't parse). **Fix:** in **runtime/function-body** contexts the constant is now spelled `bitcast<f32>(0x..u)` from its exact IEEE-754 bit pattern (covers the standalone scalar post-process and the inline-expr builder in `src/spirv_to_wgsl.zig`, and the WGSL ConstantComposite namer's vector/struct/array component loops in `src/spirv_cross_common.zig`). Because `bitcast` is itself **rejected by naga in a const-expression** (module-scope `const`, `override` spec-constant default), a non-finite float in those contexts is genuinely unrepresentable and now **honest-errors** (`error.UnsupportedOp`) rather than emitting non-parsing output — caught by the adversarial review. New naga-validated regression tests (`tests/wgsl_tests.zig`): the function-body case emits the bitcast and validates; the const-global and override cases honest-error. Adversarial review: REQUEST-CHANGES → the const-expr [Critical] + spec-const [Major] were resolved with the honest-error guards. Verified WGSL 146 / HLSL 802 / GLSL 134 / MSL 145 / opt 21 / correctness 32; conformance **PASS 2087 / FAIL 0**. **Out of scope (narrow follow-up):** an unfolded constant `1.0/0.0` is emitted as a division naga separately const-evals to inf.
- **WGSL: scalar `isinf()` now lowers to a naga-valid idiom instead of honest-erroring (#170 WGSL-coverage increment; naga-gated).** WGSL has no `isInf` builtin and no infinity literal, so the WGSL backend previously honest-errored on every `OpIsInf` ("WGSL has no isInf builtin"). Scalar `isinf(x)` now emits the literal-free idiom `(x != 0.0 && x * 2.0 == x)`, which is provably equal to IEEE `isinf` across all f32 classes: `0`/`-0` are excluded by `x != 0.0`; a finite nonzero `x` has `x*2 != x`; NaN fails the ordered `==`; and the max finite value overflows under `*2.0` to `inf`, which `!= x` (the load-bearing case). Vector `isinf` (bvecN) stays an honest-error (no componentwise `&&` on bool vectors), parallel to vector `isnan`. New naga-validated regression test (`tests/wgsl_tests.zig`) plus the pre-existing scalar isnan/isinf test flipped from asserting the honest-error to asserting the idiom (and a latent test-only spirv leak in its isinf block fixed). Adversarial review: APPROVE with a per-float-class correctness proof. Verified WGSL 143 / HLSL 802 / GLSL 134 / MSL 145 / opt 21 / correctness 32; conformance **PASS 2087 / FAIL 0**. #170 remains open (ongoing parity grind).
- **Frontend: a `barrier()`/atomic/store hidden inside a CALLED helper did not invalidate the caller's load cache → stale shared/SSBO read after the call (#234, all backends; follow-up to #223).** `void bump(){ atomicAdd(s_counter,1u); barrier(); } … uint a = s_counter; bump(); uint b = s_counter;` read `b` from the PRE-call cached load of `s_counter` — the #223 memory-ordering violation, one call-frame deep. glslpp emits a real `OpFunctionCall` (no inlining at semantic-analysis time) and has no cross-function memory-effect summary, so the callee's barrier/atomic/store — and any mutation of an out/inout pointer arg — was invisible to the caller's per-call-frame `load_cache`/`global_load_cache` (`src/semantic.zig`). Disassembly confirmed `result.total = <pre-call load>` (no-opt) and the same after the optimizer inlined the helper. **Fix:** the single user-`OpFunctionCall` emission site now clears both load caches (`invalidateLoadCachesAtBarrier`) immediately after the call — conservatively, since a callee may run any memory op. Over-invalidation only forces an extra re-load (never a stale value; `load_cache` is per-block, and immutable-uniform reloads are re-merged by the optimizer's load CSE); out/inout pointee reads correctly re-load because `analyzeLValue` materializes a real backing store for the passed pointer. After the fix the post-call read is a FRESH `OpLoad` positioned after the call (no-opt) / after the inlined atomic (opt). New frontend regression tests (`tests/optimizer_tests.zig`, opt + no-opt). Adversarial review: APPROVE (single emission path, out/inout coverage, and over-clear safety all confirmed). Verified HLSL 802 / GLSL 134 / MSL 145 / WGSL 142 / correctness 32 / opt 21 / loop-phi 15; conformance **PASS 2087 / FAIL 0** (no regression from the broad blast radius of clearing at every user call).
- **Frontend: a buffer load through one access chain was reused after an atomic/store/`++` mutated a SIBLING access chain into the SAME base — yielding a STALE pre-mutation value when the indices dynamically alias (#235, all backends; follow-up to #223).** `uint x = b.data[j]; atomicAdd(b.data[i], 1u); uint y = b.data[j];` reused `b.data[j]`'s cached load across the atomic on `b.data[i]` because the two access chains have distinct SSA ids and the load cache (`load_cache`/`global_load_cache` in `src/semantic.zig`) only invalidated the EXACT mutated chain id plus its base — not sibling chains into the same base. When `i == j` at runtime the read of `y` was the wrong (pre-atomic) value on **every** backend. **Root cause:** `invalidatePtrLoadCache` (atomics), `emitStore`, and the assignment/`++` store paths each removed only `ptr_id` + its `ac_result_to_base` base. **Fix:** new `invalidateAliasingChains(base_id)` helper iterates `ac_result_to_base` (per-block, so bounded) and drops every cached load whose chain resolves to the same base; wired into all four mutation sites (atomic RMW, `emitStore`, inline assignment store, pre/post-increment store) — each call runs BEFORE the exact-chain store-to-load-forwarding `put`, so the mutated chain still forwards its new value while siblings are invalidated. Over-invalidation is always safe (forces an extra re-load, never reuses a stale value; immutable-uniform reloads are re-merged by the optimizer's load CSE). After the fix the `b.data[j]` read is re-loaded AFTER the `OpAtomicIAdd`/store (verified by disassembly). New frontend regression tests (`tests/optimizer_tests.zig`, opt + no-opt, for both the atomic and the `b.data[i]++` forms — the increment site was caught by the adversarial review as the same bug class). Verified HLSL 802 / GLSL 134 / MSL 145 / WGSL 142 / correctness 32 / opt 19; conformance **PASS 2087 / FAIL 0**. **Deferred (pre-existing, narrower residual, tracked as a follow-up):** a sibling load cached in a *dominating* block (entry/loop-header `global_load_cache`) under a different id, and a *multi-level* nested lvalue (`b.arr[i].x` vs `b.arr[j].y`) whose chains resolve to different intermediate bases, are not covered by the single-level base match.
- **GLSL/HLSL/MSL: a `do`-while whose BODY had its own conditional control flow crashed (GLSL stack-overflow) or silently miscompiled (HLSL/MSL inverted+infinite) (#244, all three backends).** A `do { … if(x) continue; … } while(cond);` was mis-handled: the do-while's `LoopMerge` is followed by an unconditional `OpBranch` into the body, but the body's FIRST `OpBranchConditional` (the body's own `if`) was grabbed by the top-test "pattern A" scan as if it were the *loop* condition (the real condition is the back-edge `OpBranchConditional` in the continue block). Correct at arbitrary nesting depth (3-/4-level verified), for mixed struct+scalar members, and propagates `@interpolate(flat)` to integer/flat leaves; a matrix/array leaf at a `@location` is the unrepresentable case and honest-errors (`error.UnsupportedOp`). **Single-level interface blocks (no struct member) are byte-identical** — the new path is gated on `nested_io_block_types`, empty for them (`struct-varying.legacy.frag` unchanged; adversarial-review + 11-shader stress-probe confirmed). **Deferred (tracked, not a regression):** the symmetric nested-struct OUTPUT block (`out Blk { Mid m; }`) still naga-rejects — it flows through the one-level-only `io_block_outputs` loop and needs the store-side analogue. naga-gated test asserts the leaf params + reassembly + naga validity.
- **WGSL coverage deepening, Pass 15 (G5, #170):** sub-bucket H (array/nested-struct OUTPUT block sub-case) — a vertex `out` interface block with an aggregate member is now flattened into per-leaf `@location` members and reassembled, naga-gated (corpus shader `struct-flatten-inner-array.legacy.vert` flips naga REJECT → PASS; conformance unchanged at PASS 2087 / FAIL 0):
  - **H — a vertex `out` interface block whose member is an aggregate (array/struct) emitted that aggregate at a single `@location`:** `out Foo { float a[4]; }` → `@location(0) a: array<f32, 4>`, which WGSL forbids (naga: "only numeric scalars and vectors are allowed"). The array is written by a dynamic-index loop (`a[i] = …`) that cannot target separate `@location` members directly, so a store-splitting chokepoint (as used for matrix outputs) is impossible. Instead — symmetric to the Pass 14 INPUT reassembly, but in the WRITE direction — the block's scalar/vector LEAVES are collected recursively (`collectOutputLeaves`: an array expands per element, a struct member recurses) and emitted as consecutive flattened `@location` members of `VertexOutput` (`@location(0) a_0: f32 … @location(3) a_3: f32`); the original (nested) struct is **reassembled into a mutable local** (`var io_foo: Foo;`) that the body writes normally (`io_foo.a[i] = …`); and each leaf is **copied out** into its flattened member just before `return vertex_out` (`vertex_out.a_0 = io_foo.a[0];`). A **matrix member** or an **array-of-aggregate** at a `@location` is the unrepresentable case and honest-errors (`error.UnsupportedOp`). **Gated precisely:** the path only fires for an output block with an aggregate member — a plain scalar/vector output block keeps the one-per-`@location` form, and matrix outputs / single-output shaders are byte-identical (adversarial-review + 12-shader stress-probe confirmed; correct location numbering with no cross-output overlap, array-of-struct + mixed-member depth verified). naga-gated tests: the array fixture (per-element leaves + copy-out + naga) and a nested-struct output block (recursive leaves). **Pre-existing, out of scope (tracked separately):** the WGSL vertex-struct lowering collapses all returns into one exit, so a mid-body **early `return`** drops its writes (silent-wrong) — present on the plain-output path before this change and neither caused nor amplified here.

- **WGSL coverage deepening, Pass 17 (G5, #170):** sub-bucket I (spec-constant-sized arrays) — unrepresentable spec-const array sizes now honest-error, naga-gated (corpus shaders `spec-constant-work-group-size.vk.comp` + `spec-constant-op-member-array.vk.comp` flip naga REJECT → honest-unsupported; conformance unchanged at PASS 2087 / FAIL 0):
  - **I — a spec-constant-sized array (`int a[spec_const]`) is unrepresentable in WGSL except as a `var<workgroup>` type:** WGSL `override`-expression array sizing is allowed ONLY for workgroup variables, never function-locals, struct members, or storage members. glslpp emitted a runtime `array<T>` for a spec-const-sized function-local (naga: "Local variable invalid") or dropped spec-const-sized struct members entirely, yielding an empty `struct {}` (naga: "Type invalid"). The WGSL backend now fails loud (`error.UnsupportedOp`, "WGSL cannot size a non-workgroup array by a specialization constant") via a pre-pass (`typeContainsSpecConstArray`) that flags any non-`Workgroup` OpVariable whose type (recursively, through pointer/array/struct) contains an OpTypeArray with an `OpSpecConstant`/`OpSpecConstantOp` length. - **MSL whole-array VALUE ops emitted illegal Metal (#168):** a whole-array `OpLoad`/`OpStore` — e.g. a value copy `float local[N] = LUT;` — was typed by `mslType()`, which drops the `[N]` and returns only the element type, producing `float v14 = v3;` (a scalar-from-array load) and `v13 = v14;` (a C-array whole-copy — illegal, Metal C-arrays aren't assignable). The backend now uses the `spvUnsafeArray<T,N>` template as the array **value** type at value-context sites, emitted once and gated on need — mirroring `spirv-cross --msl`. Each fix is oracle-gated against `spirv-cross --msl` for structural equivalence. **Now supported (legal `spvUnsafeArray<T,N>` everywhere):**
  - whole-array `OpLoad` value copy from a const global (`float local[N] = LUT;`) — the headline case;
  - **local→local** whole-array copy — BOTH the source AND destination function locals are spelled `spvUnsafeArray` (the source-local was previously left a C-array → illegal copy);
  - **`OpSelect`/ternary** on whole arrays (`float la[N] = cond ? A : B;`) — the Select result and its dest are `spvUnsafeArray` (was a scalar-from-array Select);
  - **`OpCompositeConstruct`** of an array (`float arr[N] = float[](a,b,c);`) — emits `spvUnsafeArray<T,N>({ a, b, c })` (was a bogus `float(a,b,c)` scalar ctor + C-array copy);
  - **struct-element** const-array value copy (`const S LUT[N]; S local[N]=LUT;`) — `struct S` is now declared **before** the `constant spvUnsafeArray<S,N>` that references it (was use-before-declaration).
  - The read-only index-only path keeps the plain `constant T[N]` spelling (intentional, valid divergence). The `.Load` template spelling is narrowed to loads whose source is itself an `spvUnsafeArray` (a value-copied const or value-copied local); a whole-array load from any other source now **fails loud** (`UnsupportedWholeArrayValueLoad`) instead of emitting a template/​C-array mismatch.
  - **Honest-errored (no silent-wrong), deferred to a frontend fix — see #173:** a **spec-constant-sized** array used as a whole-array value (`mslValueType` cannot read a literal length → `UnresolvableArrayLength`, was silently sized `<T,1>`); a **matrix-element** const-array global (`const mat4 M[N]`) that the frontend does not fold to an `OpConstantComposite` → `UndeclaredPrivateArrayGlobal` (was a reference to an undefined `M` identifier). Nested multi-dimensional const-array globals copied whole likewise still depend on a separate, pre-existing frontend folding bug; out of scope here.
- **Spec-constant array indexed by a runtime expression produced dangling SPIR-V:** a constant evaluated at top level (spec-constant default / global `const`) was cached but its instruction discarded, so a later function referenced an unemitted constant (`OpISub %<dangling> %a`) → invalid SPIR-V with exit 0. The top-level constant cache is now cleared before function analysis. (`spec-constant-work-group-size.vk.comp` now passes; strict-gate restored to green.)
- **GLSL integer fragment inputs missing `flat`:** glslang requires `flat` on integer/double fragment inputs; the backend dropped the `Flat` decoration → glslang-rejected output. Now emitted faithfully (fragment inputs and flat vertex outputs).
- **GLSL multidimensional array locals dropped a dimension** (`vec4 v[2];` for a `vec4[2][2]` value) → glslang type-mismatch on assignment. The local-var array suffix now walks all nested dimensions.
- **Float-vector constructor sign flip:** `vec2(2147483648, 0)` folded the bare (signed) integer literal with the wrong sign (`+2.147e9`); glslang wraps 2³¹ to `-2147483648` → `-2.147e9`. Now matches glslang; `u`-suffixed literals stay unsigned.
