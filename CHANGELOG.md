## [Unreleased]

### Fixed (silent-wrong elimination, all oracle-gated)
- **Two `OpTypeImage` codegen bugs: per-image storage `Format` dropped with >1 image, and `samplerCubeArray` emitted as a plain cube (#183, silent-wrong, glslangValidator-gated).** Both surfaced via the #177 reflection serializer (the JSON faithfully read back glslpp's own decorations; on glslang-built SPIR-V the output matched the spirv-cross oracle, so these were CODEGEN defects).
  - **Storage-image `Format` dropped when more than one storage image is present.** The type dedup in `src/codegen.zig` keyed storage images by `@intFromEnum(ty)` ALONE — format-blind — so a second `image2D` of a different `layout(rgbaN)` qualifier reused the first one's `OpTypeImage` and was silently decorated with the first image's `Format` (e.g. a `rgba32f` image emitted as `rgba8`). An array-of-images bypassed the pre-pass entirely and resolved its element via the format-blind `ensureType(.array)→ensureType(.image2d)`, emitting `Unknown`. **Root cause = the dedup/threading, not the (already-correct) `emitStorageImageType`.** Fixed by (a) a dedicated format-aware cache `emitted_storage_image_types` keyed by `(enum, format)`; (b) recording each storage-image global's declared format keyed by its variable result id (`storage_image_format_by_ptr`) and threading it through the OpVariable pointer type, the array element type, the array-element access chain, AND the `.load` result type — so each image keeps its own format even when several share the same GLSL enum. Sampled images (always `Unknown` format) are deliberately left on the format-blind dedup. Now matches `glslangValidator -V` exactly: two distinct images emit both `Rgba8` AND `Rgba32f`; an `rgba32f image2D arr[3]` emits an `Rgba32f` element; the 40-format `image-formats` fixture's per-image formats are identical to the oracle multiset.
  - **`samplerCubeArray` emitted as a plain cube (`Arrayed=0`).** `src/parser.zig` mapped `kw_sampler_cube_array → .sampler_cube` (there was no float cube-array AST variant — only the int/uint `isampler/usampler` siblings and `samplerCubeArrayShadow` existed). The emitted `OpTypeImage` therefore set `Arrayed=0`, spelling/behaving as `samplerCube` and losing the array dimension. Fixed by adding a `sampler_cube_array` AST variant (`src/ast.zig`) threaded through every required switch (`isSampler`/`isCombinedSampler`/`samplerBaseType→.float`/`scalarSize`/`numComponents`), the parser, a new codegen arm modeled on `isampler_cube_array` but with base `.float` (`Cube, Depth=0, Arrayed=1, MS=0, Sampled=1, Unknown`) plus the cube-array capability/cube lists and the `texture()` inner-image resolution, and `src/semantic.zig` arg-validation/result-typing (`textureSize`→`ivec3`, `texture()` `vec4` coord via `samplerResultType`). Now emits `%float Cube 0 1 0 1 Unknown` (`Arrayed=1`), matching the oracle and the already-correct `isamplerCubeArray`; the variable is also now correctly classified `UniformConstant` (was wrongly `Uniform`), so `spirv-cross --reflect` agrees the type is `samplerCubeArray`. (HLSL correctly upgrades to `TextureCubeArray`; the MSL/WGSL/GLSL backends still emit the non-array cube name — a PRE-EXISTING backend gap shared by `sampler2DArray`, out of #183's SPIR-V-codegen scope.)
  - Failing-first glslang-gated tests in `src/gap_tests.zig` (walk `OpTypeImage` Format/Arrayed operands); the #177 reflection tests (`tests/reflection_tests.zig`) tightened to the now-correct oracle values (array-image element `rgba8`→`rgba32f`, a 2nd distinct `rgba32f` scalar storage image asserting two distinct formats, and a `samplerCubeArray` binding asserting the `samplerCubeArray` spelling). `spirv-val` clean on all new fixtures; conformance **PASS 2081 / FAIL 0** (up 1 — `tests/spirv-cross/image-query.desktop.frag`, previously KNOWN_UNSUPPORTED only because `samplerCubeArray`/`imageCubeArray` were mis-compiled, now compiles + validates and is delisted in `tests/runner.zig`).
  - **#183 review — `samplerCube` + `samplerCubeArray` coexistence emitted invalid SPIR-V (BLOCKER).** A single `sampled_image_cube_inner_id` field in `src/codegen.zig` was written by ALL FOUR cube `ensureType` arms (`.sampler_cube` Arrayed=0, `.sampler_cube_array` Arrayed=1, `.sampler_cube_shadow` Arrayed=0, `.sampler_cube_array_shadow` Arrayed=1), so whichever ran last clobbered it. The `extract_image` (`OpImage`) site mapped all four types to that one field → an `OpImage`/`textureSize` on the *other* cube type referenced the wrong-`Arrayed` inner image type → `spirv-val`: "Expected Sample Image image type to be equal to Result Type". Newly reachable on this branch (the Bug-2 fix made `samplerCubeArray` a distinct `Arrayed=1` type; on `main` it had collapsed to `.sampler_cube`). Fixed by splitting the inner id into four distinct fields — `sampled_image_cube_inner_id`, `sampled_image_cube_array_inner_id`, `sampled_image_cube_shadow_inner_id`, `sampled_image_cube_array_shadow_inner_id` (also fixing the **pre-existing** non-array-vs-shadow cube collision) — each set in its own `ensureType` arm, and selecting the field by `inst.ty`'s arrayed-/shadow-ness at the extract site. A shader with both a `samplerCube` and a `samplerCubeArray` (each with a `textureSize` extraction) now emits two distinct `OpTypeImage` (`Cube … 0 …` and `Cube … 1 …`) with each `OpImage` referencing its correct-`Arrayed` inner type, `spirv-val`-clean.
  - **#183 review — the delisted `image-query.desktop.frag` passed only via DCE masking (MAJOR).** Its body is entirely dead, so the optimizer stripped every `OpImage` extraction; it spirv-val'd because the buggy ops were *gone*, not correct. Added an escaping-DCE coverage test in `src/gap_tests.zig` ("samplerCube + samplerCubeArray coexist with correct-Arrayed OpImage") that writes both `textureSize` results into `gl_FragColor`, so the two `OpImage` extractions survive optimization, and asserts they reference image types with distinct `Arrayed` bits (0 and 1). Fails pre-BLOCKER-fix, passes after. `image-query.desktop.frag` stays delisted (it is valid) but is no longer the coverage for this fix.
- **Codegen emitted the WRONG std140/std430 `Offset` for a member following a nested struct (#181, silent-wrong, glslang-gated):** the std140/std430/scalar layout helpers (`layoutSize`, `layoutAlignmentStd140`, `layoutAlignmentStd430`, `layoutAlignmentScalar` in `src/codegen.zig`) looked a nested struct up only in `self.emitted_named_types`. But a struct reached through a UBO/SSBO interface block is cached in `self.emitted_interface_named_types` (see the `in_interface_block` split in `ensureType`). The lookup missed and returned the silent-wrong defaults — `layoutSize` → 0 (so the next member never advanced and OVERLAPPED the struct), alignments → 16/8. This corrupted: the `Offset` of any member after a nested struct or array-of-struct, the `ArrayStride` of struct arrays, and the alignment after a small std430 struct. Example: std140 `uniform Scene { Material mat; mat4 mvp; }` (where `Material{vec4 albedo; Light l;}`) emitted `mvp Offset 0` (overlapping `mat`) where glslang emits `Offset 48`. Fixed with the same dual-lookup fallback (`emitted_interface_named_types` first, then `emitted_named_types`) already used by the nested-struct recursion sites, plus removing the struct from the `layout_visited` cycle set before computing its own alignment (the self-ref guard otherwise over-rounded a small struct's size). A both-maps-miss path now `std.log.warn`s instead of silently returning 0. New glslang-gated offset-comparison tests (`tests/std140_nested_offset_tests.zig`, `zig build test-std140-nested-offset`) and a conformance fixture (`nested_struct_block.frag`); the #177 reflection tests previously pinned to the buggy values are tightened to the oracle (block_size 64→112, mvp.offset 0→48, block_size 96→100, intensity.offset→96, runtime-array-of-struct ArrayStride 0→32). Non-nested layouts unaffected (`test-std430-matrix-layout`/`test-scalar-layout`/`test-buffer-ref` green); conformance **PASS 2080 / FAIL 0**. **Follow-up (#181 review):** `layoutAlignmentStd140` for a struct now rounds the max-member alignment **UP to a multiple of 16** (vec4 alignment) per the std140 rule, so a non-16-aligned member following a scalar-only or vec2-only nested struct lands at the correct offset — e.g. std140 `uniform U { S s; float tail; }` with `S{float a}` now emits `tail Offset 16` (was 4), and `V2{vec2 a}` likewise emits 16 (was 8), matching glslang. The struct's std140 size rounds to the now-≥16 alignment automatically (`layoutSize`). **std430 is deliberately NOT rounded to 16** (`layoutAlignmentStd430` keeps a struct's real max-member alignment — a `struct{float}` in std430 stays 4-aligned, so `t Offset 4`); a glslang-gated std430 control case proves this. The repeated `emitted_interface_named_types.get(name) orelse emitted_named_types.get(name)` pattern (6 sites) is extracted into a `resolveLayoutTypeId` helper (behavior identical; each site keeps its own miss-default).
- **Reflection JSON mis-spelled array / shadow samplers and dropped image/spec-const fidelity (#177 review, all oracle-gated against `spirv-cross --reflect`):** four fidelity gaps in the #177 JSON serializer (`src/reflection.zig`), the feature's whole purpose:
  - **`sampler2DArray` / `sampler2DShadow` were mis-spelled `sampler2D` (MAJOR, silent-wrong):** `spvImageName`/`spvSampledImageName` keyed only off the `OpTypeImage` `Dim` operand; the `depth` and `arrayed` operands were parsed in a comment but never stored on the internal `TInfo`. The serializer therefore emitted `sampler2D` for `sampler2DArray`, `sampler2DShadow`, and `sampler2DArrayShadow` alike. `depth`/`arrayed` are now stored at the `OpTypeImage` parse and the spelling appends `Array` (arrayed==1) then `Shadow` (depth==1) per spirv-cross — `sampler2DArray` / `sampler2DShadow` / `sampler2DArrayShadow` now match the oracle exactly.
  - **`images` section omitted `format` and `array_size_is_literal` (MINOR):** the storage-image binding section emitted `{"type":"image2D"}` without the `"format":"rgba8"` (though `Resource.image_format` was populated) and `"array":[3]` without the paired `"array_size_is_literal":[true]`. Both are now emitted, matching the oracle.
  - **Spec-constant `default_value` was raw u32 bits without a `type` (MINOR):** float `2.5` serialized as `1075838976`. The serializer now emits a `"type"` field (`float`/`bool`/`int`/`uint`, via a new `OpTypeBool` parse + `spvTypeName` on the spec-constant's type) and decodes `default_value` per that type (`2.5` / `true` / `7`).
  - **Anonymous block resource emitted `"name":""` (MINOR):** a UBO/SSBO with no instance variable name now falls back to the block TYPE name (e.g. `"Scene"`, from `Resource.type_name`) in the serializer only — `Resource.name` is unchanged.
  - **NIT:** the silent-wrong fallbacks in `spvTypeName` (vector `else => "vec"`) and `spvImageName` (Dim `else => "2D"`) now surface a clearly-wrong token rather than a plausible-but-wrong spelling, and the cube/subpass cases are handled.
  - All four assertions are derived from and diffed against `spirv-cross --reflect` on glslpp's own binaries; new tests in `tests/reflection_tests.zig`. Two pre-existing **codegen** bugs surfaced (not reflection defects, faithfully reflected — the oracle on the same binary agrees): glslpp drops the per-element format on a storage-image array (and assigns the first image's format to all storage images), and spells `samplerCubeArray` as a plain cube (arrayed==0). Conformance unchanged at **PASS 2079 / FAIL 0**.
- **Codegen emitted the WRONG SPIR-V `Decoration` value for `coherent`/`volatile` buffers (#177, silent-wrong):** the `spirv.Decoration` enum had `coherent = 0` (0 is **RelaxedPrecision**, not Coherent) and `@"volatile" = 22` (22 is **NoSignedWrap**, not Volatile). The codegen `coherent`-buffer path (`codegen.zig`) therefore emitted `OpDecorate %buf RelaxedPrecision` instead of `OpDecorate %buf Coherent` — a memory-coherence qualifier was silently lost (and would, if `volatile` emission were ever wired, decorate with NoSignedWrap). Corrected to the SPIR-V spec values **Coherent = 23, Volatile = 21** (Restrict = 19 was already correct). Verified via `spirv-dis` (before: `%B_0 RelaxedPrecision`; after: `%B_0 Coherent`) with `spirv-val` still clean, and the full conformance suite still **PASS 2079 / FAIL 0** (no fixture regressed). A grep confirmed nothing else in the codebase decoded decoration value 0 or 22 expecting the old (wrong) meaning — the only consumer was the codegen emit site.
- **Optimizer deleted live result-less instructions, emitting dangling type ids (#173 review, MAJOR):** the algebraic-identity pass (`algebraicSimpl` in `compact_ids_passes.zig`) decided whether to drop an instruction by reading `words[pos+2]` as its result id for *every* instruction with word-count ≥ 3. That word is a result id only for type/result-bearing ops; for **result-less** instructions like `OpStore <ptr> <obj>`, `words[pos+2]` is the *object operand*. When an identity fold (e.g. `acc += m[0]` with `acc == vec4(0)` → `0.0 + x → x`) put that object id into the replacement map, the whole `OpStore` was silently deleted — orphaning the store's value/type chain, so a later DCE pass dropped the `OpTypeFloat` while a live `OpTypeVector`/output `OpVariable` still referenced it. Result: spirv-val "Operand '%N' requires a previous definition" on otherwise-valid input (reproduced on `mat3x4` const-array shaders, where `m[0]` is a `vec4` extract folded into the store). The skip-elimination check now derives the genuine result id position from `getOpInfo(...).fixed` (result at `pos+2` for type+result ops, `pos+1` for result-only ops) and never treats an operand of a result-less op as a removable result. Verified per-pass with spirv-val; DCE + id-compaction were confirmed correct in isolation (they retain a type referenced only by a live type), so the single root cause was the operand/result confusion. New spirv-val-gated conformance fixtures `mat3x4_const_array_fold_dce.frag` (body-folds-away) and `mat3x4_const_array_fold_live.frag` (body survives), plus a cross-compare regression test.
- **Diagonal matrix constructor lost constant CSE (#173 review):** the #173 diagonal-ctor path (`mat4(1.0)`) was switched from `emitPureOp` to an explicit append + `tryUpgradeToConstantComposite` (needed so the upgraded column is a real `constant_composite`), but it no longer consulted `const_composite_cache`, so `mat4(1.0)` used twice emitted duplicate `OpConstantComposite`s (valid but bloated — 11 vs 6 in the no-opt frontend output). It now checks the cache (same `constCompositeKey` the from-scalars/array-ctor paths use) before emitting each diagonal column and the final matrix, reusing the cached id on a hit.
- **Frontend const-array folding gaps (#173 items 1+2), cascading to ALL backends:**
  - **Matrix-element const-array globals were never folded → garbage/undeclared on every backend.** `const mat4 M[N] = mat4[](…)` (and any standalone `const mat4 M = mat4(…)`) never folded to an `OpConstantComposite`, because the matrix constructors themselves emitted only `OpCompositeConstruct` — so the enclosing array could not fold and the Private `OpVariable` got no initializer. Symptoms: GLSL/HLSL/WGSL read uninitialised garbage (or referenced an undefined `M`), and MSL honest-errored (`UndeclaredPrivateArrayGlobal`, from the #168 review). The matrix **diagonal** ctor (`mat4(1.0)`) and **from-scalars** ctor (`mat4(a,b,…)`) now upgrade each column AND the final matrix to `constant_composite` when all components are constant (via `tryUpgradeToConstantComposite`, mirroring the vec-splat path), so the existing array-constructor upgrade folds `mat4[](…)` automatically and the init-attach path sets `initializer_id`. The diagonal path builds its columns with an explicit append (not the cached `emitPureOp`) so the upgraded column id is genuinely a `constant_composite` recognised by `isConstantId`. **No silent-wrong remains on any backend:** GLSL emits `const mat4 v[2] = {mat4(vec4(…),…), …}` (glslang-validated), WGSL emits `const M: array<mat4x4f,2> = array<mat4x4f,2>(mat4x4f(vec4f(…),…), …)` (naga-validated), and MSL materialises `constant float4x4 name[2] = { float4x4(float4(…),…), … }`. The MSL backend's `writeMslConstInit` previously emitted nested matrix/vector constituents **by name** (`v4`, `v5`) which were never defined at module scope — it now **inlines** them as typed constructors (`float4x4(float4(…),…)`), the only backend change required.
  - **Sized multi-dimensional array constructor swapped its dimensions.** The `type_constructor` sized-array bracket loop in `parser.zig` wrapped dims inner-first, so `float[2][3](…)` produced the type `array{size=3, base=array{size=2, base=float}}` instead of the correct `array{size=2, base=array{size=3, base=float}}`. This made `const float m[2][3] = float[2][3](…)` a type error against its (correctly built) declared type, and — when folded — an `OpConstantComposite` whose constituent count did not match its `OpTypeArray` length (spirv-val "Constituent count does not match NumElements"). The loop now collects all bracket dims in source order (preserving `0` for `[]`) then reverse-wraps outermost-to-innermost, mirroring the already-correct declaration-side logic. Only the SIZED multi-dim form changed; the unsized `float[][](…)` path and the single-dim form are byte-identical. spirv-val-gated test plus a new conformance fixture (`sized_multidim_ctor.frag`).
  - **Item 3 retained as the documented contract:** a **spec-constant-sized** array used as a whole-array value (`mslValueType` cannot read a literal length) stays an honest `error.UnresolvableArrayLength` — symbolic-`Num` templating is deferred (test kept).
- **Reflection `block_size` undercounted for array-tail and padded-matrix-tail blocks (#171 review):** the initial #171 implementation derived `block_size` as `last_member.offset + last_member.size`, but `last.size` is the array ELEMENT byte_size (set at `OpTypeArray`) or a matrix's `colvec*cols` — neither accounts for array length×stride or `mat3` column padding. A std140 UBO ending in `float tail[4]` reported 20 instead of the oracle's 80; `float md[2][3]` reported 20 vs 112; a trailing `mat3` reported 52 vs 64; a std430 SSBO ending `float tail[5]` reported 20 vs 36. `block_size` is now `max(member.offset + member_extent)` where `member_extent` is `ArrayStride*length` for sized arrays (the OUTER stride spans inner dims for multidim), `MatrixStride*columns` for matrices, 0 for runtime arrays, and `member.size` otherwise — all formulae matched against `spirv-cross --reflect`. (Existing tests passed only because their fixtures happened to end in a scalar or `mat4`, where the old formula coincidentally matched.)
- **WGSL coverage deepening, Pass 1 (G5, #170):** three classes of WGSL backend defects, all naga-gated:
  - **`findMSB(uint)` was a needless honest-error:** GLSL `findMSB` on an unsigned operand emits GLSL.std.450 **FindUMsb (75)**, which was missing from the shared name table → `error.UnsupportedExtInst` for an op WGSL fully supports. FindUMsb/FindSMsb/FindILsb now map directly to `firstLeadingBit`/`firstTrailingBit` in the single-source-of-truth table (retiring a dead main-path-only remap). GLSL findMSB/findLSB always return *signed* int even for an unsigned operand, while WGSL's bit-scan builtins return the argument's type, so the result is wrapped in an explicit `rt(…)` conversion (identity cast when types already match) in both the main and loop/switch-replay ExtInst paths. Also added the NaN-variants **NMin (79) → `min`, NMax (80) → `max`, NClamp (81) → `clamp`** (spirv-cross mappings, naga-validated).
  - **Projective texture sampling is now lowered dimension-aware (non-Dref), Dref honest-errored:** `textureProj*` (ImageSampleProj{Implicit,Explicit,DrefImplicit,DrefExplicit}Lod) previously lowered to a hard-coded `textureSample(t, s, coord.xy / coord.w)` — correct *only* for the 2D/vec4 case, but wrong for vec3 coords (the divisor must be the coordinate's LAST component, `.z`, not `.w`) and for 3D/cube/Dref forms. Pass 1 over-corrected it to a blanket honest-error, which regressed the working 2D case. The non-Dref forms (ImageSampleProjImplicitLod/ExplicitLod) now lower **dimension-aware**: the coordinate is divided by its actual last component (read from the operand vector width) and the leading components match the sampler dimensionality (`.x` 1D / `.xy` 2D / `.xyz` 3D), producing a naga-validated `textureSample`/`textureSampleLevel`. The projective **depth-compare** forms (ImageSampleProjDref{Implicit,Explicit}Lod) and **cube/arrayed** projective forms have no faithful WGSL mapping and stay an honest `error.UnsupportedOp` ("WGSL has no projective depth-compare sampling" / "… for this sampler kind") rather than silently dropping the compare. (The ExplicitLod path mirrors the non-proj arm but is not yet reachable through glslpp's GLSL frontend, which rejects `textureProjLod` itself — see deferred note.)
  - **Fragment-shader interlock barriers silently dropped:** `OpBeginInvocationInterlockEXT`/`OpEndInvocationInterlockEXT` were unhandled (dropped). They now honest-error with `error.UnsupportedOp` as defense-in-depth (the interlock *execution mode* was already caught).
  - **The main-path `else` placeholder was silent-wrong:** the fallback emitted `// unhandled op N` + an uninitialised `var <name>: T;` (a garbage value that naga nonetheless accepts as valid syntax). It now **fails loud** with `error.UnsupportedOp` ("WGSL: unsupported op …"), mirroring the loop/switch-replay `else` which already did. Verified safe: a grep for `"unhandled op"` over the full conformance corpus output is empty, so nothing representable relied on the placeholder.
- **WGSL coverage deepening, Pass 2 (G5, #170):** matrix/literal correctness + two silent-wrong AUDIT FIXES, all naga-gated (corpus `naga PASS` rose 1945 → 1960, `REJECT` 34 → 27, `honest-unsupported` 64 → 56; conformance unchanged at PASS 2079 / FAIL 0):
  - **`inverse()` (GLSL.std.450 MatrixInverse 34) now lowers to a generated `spvInverseN` helper** instead of honest-erroring. WGSL has no `inverse` builtin (naga: "no definition in scope"). A new emit-once mechanism (threadlocal `needs_inverse_{2,3,4}` flags set by a pre-emit scan, helpers written into the module preamble before any body — mirroring the spirit of the MSL `spvUnsafeArray` template injection) emits closed-form cofactor/determinant `spvInverse2`/`spvInverse3`/`spvInverse4` for the square `mat2`/`mat3`/`mat4` cases. Each helper is naga-validated. Non-square / unsupported sizes have no inverse and stay an honest `error.UnsupportedExtInst`.
  - **`outerProduct()` (OpOuterProduct) now constructs the matrix explicitly** (was the honest-error else-arm). `outerProduct(u, v)` builds a `matCxR` whose column *i* is `u * v[i]` (`v`'s component count columns of `u`'s component count rows), emitted as `matCxRf(u*v.x, u*v.y, …)`. Added to BOTH the main emit dispatch and the switch/loop-replay dispatch. naga-validated for square (mat2/mat3) and non-square (mat2x4).
  - **Abstract scalar-literal typing:** naga rejects all-constant-arg builtin calls (e.g. `smoothstep(0.08, 0.03, 1.0)`) with "Abstract types may only appear in constant expressions". Scalar `TypeFloat` constant literals are now suffixed with `f` (`0.08` → `0.08f`) in the WGSL constant-name post-process — scalar-float only (the composite path already emits concrete `vec3<f32>(…)`, and typing bare abstract ints with `i` regressed correct already-passing output such as `textureGather` component indices, so it was intentionally NOT applied).
  - **AUDIT FIX — subgroup ops were SILENT-WRONG:** all twelve+ subgroup arms (`subgroupElect`/`Ballot`/`Broadcast`/`BroadcastFirst`/`All`/`Any`/`Shuffle*`/`Inclusive*`/`Exclusive*` + scan/reduction forms) emitted WGSL subgroup builtins with NO `enable subgroups;`. naga 29.0.3 rejects subgroups entirely (even `enable subgroups;` is unsupported), so glslpp exited 0 on output naga rejects. All now `error.UnsupportedOp` ("WGSL/naga does not support subgroup operations").
  - **AUDIT FIX — image atomics were SILENT-WRONG:** an `imageAtomic*` lowered to `atomicAdd(&textureLoad(img, …))`, which naga rejects ("operand of `&` must be a reference") — WGSL has no image atomics. The `OpAtomic*` handlers (`emitAtomicBinOp` + the inline `AtomicExchange`/`AtomicCompareExchange` arms) now detect an `OpImageTexelPointer` pointer operand and `error.UnsupportedOp` ("WGSL has no image atomic operations") instead. Buffer/workgroup atomics on a real pointer are unaffected.
- **WGSL coverage deepening, Pass 3 (G5, #170):** push-constant blocks + one silent-wrong AUDIT FIX, all naga-gated:
  - **A1 — push-constant blocks were dropped → dangling references (silent-wrong):** in the global-variable collection loop of `src/spirv_to_wgsl.zig`, the storage-class `switch` handled `.Uniform`/`.StorageBuffer`/`.UniformConstant` but `.PushConstant` fell into the `else => {}` arm, so the `PushConstants` struct type AND the `push` var were never emitted while the body still referenced `push.value0` → naga "no definition in scope for push". WGSL has NO push_constant address space (naga 29.0.3 rejects both `var<push_constant>` and `enable push_constant`); the representable lowering is a plain **uniform buffer**. A `.PushConstant =>` arm now mirrors `.Uniform`, appending the block to `cbuffers` (`is_ssbo=false`); push constants carry no Binding/DescriptorSet decoration so they default to binding 0, and the existing binding-dedup pass bumps it if a real UBO already owns 0. All downstream machinery (struct forward-decls, name-collision rename, `push.value0` access chains) is reused for free — a vec4-member push-constant block now fully naga-validates (`struct PushConstants {…}` + `var<uniform> push: PushConstants`). Blocks with a scalar-array member (`float Arr[4]`) still hit the shared array-stride limitation, but A1 converts them from "no definition in scope" to the honest array-stride error (incremental progress; the per-member `array<vec4f,N>` + `.x`-injection fix is deferred to Pass 4 as item A2).
  - **AUDIT FIX (item H) — texel buffers were SILENT-WRONG:** an `OpTypeImage` with `Dim=Buffer` (GLSL `samplerBuffer`/`imageBuffer`) was spelled `texture_buffer<f32>`, which is NOT a real WGSL type (naga rejects it). WGSL has no texel-buffer type, so `wgslType` now `error.UnsupportedOp`s ("WGSL has no texel buffer / texture_buffer type") for `Dim=5`, covering both the sampled and storage texel-buffer paths, instead of emitting a silent-wrong-shaped type name.
- **Module-scope `const` array globals read uninitialised memory (all backends):** a `const T arr[N] = T[](…)` global indexed by a *runtime* value lowered to a Private `OpVariable` with no initializer and no stores — its values appeared nowhere in the SPIR-V, so every backend read garbage. The frontend now folds the initializer to an `OpConstantComposite` and emits it as the variable's initializer (spirv-val clean). **GLSL** (`const T v[N] = {…}`, glslang-validated), **HLSL** (`static const`, dxc-validated), and **WGSL** (`const v: array<T,N> = array<T,N>(…)`, naga-validated) materialise the values; **MSL** indexes the read-only const array via a program-scope `constant T name[N] = {…}` (valid Metal, simpler than spirv-cross's always-`spvUnsafeArray`).
- **MSL whole-array VALUE ops emitted illegal Metal (#168):** a whole-array `OpLoad`/`OpStore` — e.g. a value copy `float local[N] = LUT;` — was typed by `mslType()`, which drops the `[N]` and returns only the element type, producing `float v14 = v3;` (a scalar-from-array load) and `v13 = v14;` (a C-array whole-copy — illegal, Metal C-arrays aren't assignable). The backend now uses the `spvUnsafeArray<T,N>` template as the array **value** type at value-context sites, emitted once and gated on need — mirroring `spirv-cross --msl`. Each fix is oracle-gated against `spirv-cross --msl` for structural equivalence. **Now supported (legal `spvUnsafeArray<T,N>` everywhere):**
  - whole-array `OpLoad` value copy from a const global (`float local[N] = LUT;`) — the headline case;
  - **local→local** whole-array copy — BOTH the source AND destination function locals are spelled `spvUnsafeArray` (the source-local was previously left a C-array → illegal copy);
  - **`OpSelect`/ternary** on whole arrays (`float la[N] = cond ? A : B;`) — the Select result and its dest are `spvUnsafeArray` (was a scalar-from-array Select);
  - **`OpCompositeConstruct`** of an array (`float arr[N] = float[](a,b,c);`) — emits `spvUnsafeArray<T,N>({ a, b, c })` (was a bogus `float(a,b,c)` scalar ctor + C-array copy);
  - **struct-element** const-array value copy (`const S LUT[N]; S local[N]=LUT;`) — `struct S` is now declared **before** the `constant spvUnsafeArray<S,N>` that references it (was use-before-declaration).
  - The read-only index-only path keeps the plain `constant T[N]` spelling (intentional, valid divergence). The `.Load` template spelling is narrowed to loads whose source is itself an `spvUnsafeArray` (a value-copied const or value-copied local); a whole-array load from any other source now **fails loud** (`UnsupportedWholeArrayValueLoad`) instead of emitting a template/​C-array mismatch.
  - **Honest-errored (no silent-wrong), deferred to a frontend fix — see #173:** a **spec-constant-sized** array used as a whole-array value (`mslValueType` cannot read a literal length → `UnresolvableArrayLength`, was silently sized `<T,1>`); a **matrix-element** const-array global (`const mat4 M[N]`) that the frontend does not fold to an `OpConstantComposite` → `UndeclaredPrivateArrayGlobal` (was a reference to an undefined `M` identifier). Nested multi-dimensional const-array globals copied whole likewise still depend on a separate, pre-existing frontend folding bug; out of scope here.
- **Spec-constant array indexed by a runtime expression produced dangling SPIR-V:** a constant evaluated at top level (spec-constant default / global `const`) was cached but its instruction discarded, so a later function referenced an unemitted constant (`OpISub %<dangling> %a`) → invalid SPIR-V with exit 0. The top-level constant cache is now cleared before function analysis. (`spec-constant-work-group-size.vk.comp` now passes; strict-gate restored to green.)
- **GLSL integer fragment inputs missing `flat`:** glslang requires `flat` on integer/double fragment inputs; the backend dropped the `Flat` decoration → glslang-rejected output. Now emitted faithfully (fragment inputs and flat vertex outputs).
- **GLSL multidimensional array locals dropped a dimension** (`vec4 v[2];` for a `vec4[2][2]` value) → glslang type-mismatch on assignment. The local-var array suffix now walks all nested dimensions.
- **Float-vector constructor sign flip:** `vec2(2147483648, 0)` folded the bare (signed) integer literal with the wrong sign (`+2.147e9`); glslang wraps 2³¹ to `-2147483648` → `-2.147e9`. Now matches glslang; `u`-suffixed literals stay unsigned.

### Added

Array-of-struct members (incl. runtime `Particle particles[];`) are now expanded so the element struct appears in `types` and is referenced by the member — lifting the item-1 array-of-struct limitation for the JSON path. Names are JSON-escaped. Implemented with a small manual writer (no `std.json` dependency for output; tests round-trip the result through `std.json.parseFromSlice` to assert validity + shape). Oracle-gated against `spirv-cross --reflect` on a nested-struct UBO, a runtime-array SSBO, and a UBO+texture shader. Additive only — codegen and the Zig-struct reflection API are unchanged; conformance unchanged at **PASS 2079 / FAIL 0**.
