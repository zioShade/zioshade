# Status Numbers — Single Source of Truth — Design Spec (2026-05-31)

> **Status:** approved design, pending implementation plan (`writing-plans`).
> **Origin:** cleanup explicitly flagged in-tree at `docs/IMPLEMENTATION_STATUS.md:181`

> 'single source of truth for status numbers' cleanup"). Layer 1 (Trust & integrity).

## Goal

Make every published test / conformance count in the docs **generated from a real run**, never
hand-edited, so they cannot silently drift from reality again.

The numbers have drifted repeatedly. Recent history: the unit-test count was stated as 1,996 →
2,054 → 2,060 across a few commits; conformance was once stated as "2,087 / 2,087 PASS (zero
failures)" when the truth is **2,080 PASS / 7 known feature-gap FAIL / 8 SKIP / 2,095 TOTAL** — and
the conformance runner even `exit(1)`s while those 7 remain (`tests/runner.zig:335`). The same
figures are restated, by hand, in at least five files and go stale independently:

| File | What it restates today (origin/main `2574b33b`) |
|---|---|
| `README.md:28,30` | conformance `2,080 / 2,087` + 7 fails + 8 skip + 2,095 total; WGSL `470/470` |
| `docs/TEST_COVERAGE.md:3,48,58,59` | conformance prose; `2,080 pass / 7 known-fail`; **hardcoded "2,054 unit tests"** |
| `docs/IMPLEMENTATION_STATUS.md:13,38,59,169` | conformance `2,080/2,087`; **stale per-suite table `1,550`… (§1.6, lines 113-128)**; **stale per-stage `1,550` pass (line 51)** |
| `CONTRIBUTING.md:33` | gate: `2,080 PASS / 7 known-FAIL / 8 SKIP` |
| `.github/PULL_REQUEST_TEMPLATE.md:13` | gate: `2,080 PASS / 7 known feature-gap FAIL` |

The conformance scalars happen to be in sync *right now* (someone hand-fixed them), but the
per-suite / per-stage / unit-test figures are already stale, which is exactly the recurring failure
mode. The fix removes hand-maintenance entirely.

## Non-goals

- **Not** wiring the conformance runner itself to XFAIL the 7 known fails (it still `exit(1)`s).
  That is the separate **analyzer-fail-loud** milestone (`docs/specs/2026-05-31-analyzer-fail-loud-design.md`).
  Here the allowlist is consumed **only** by the status generator.
- **Not** cross-platform. The runner hardcodes a Windows `spirv-val.exe` path
  (`tests/runner.zig:5`), so `conformance` is Windows + Vulkan-SDK only. `update-status` is a
  maintainer-run, Windows tool. Documented as a limitation; no skip-flags in v1.
- **Not** a new CI job. `just status` is designed to drop into CI/pre-commit later, but adding the
  hook is out of scope.
- **Not** touching the prose. Only the numbers (inside markers) and the generated `docs/STATUS.md`
  change.

## Architecture — four pieces

### 1. `docs/STATUS.md` — canonical, 100 % generated

The single dashboard. Written wholesale by the generator between a `<!-- GENERATED by
`just update-status` — do not edit by hand -->` banner. Contains:

- **Headline:** `2,080 PASS / 7 FAIL / 8 SKIP / 2,095 TOTAL` (+ runnable = pass + fail = 2,087).
- **Per-suite conformance table** (the figures currently stale elsewhere): one row per
  `=== <suite> ===` section the runner emits — `glslang-430, spirv-cross, ghostty, mesh-task,
  ray-tracing, compute, geometry, tessellation, stress` — each with pass / fail / skip.
- **Unit + HLSL totals** from the two `zig build` summaries (and, best-effort, a per-module
  breakdown from the `run test <name> N passed` lines).
- **The 7 known feature-gap fails, named**, each with its one-line reason, lifted verbatim from the
  allowlist (piece 4) — labelled *known feature-gap fails, not regressions*.
- A static "regenerate with `just update-status`" line. **No timestamp / no commit SHA** — STATUS.md
  must be byte-stable when the numbers are stable, or the drift check (piece 2, check mode) would
  fire on every commit. Provenance is the suite names + zig version as static text.

### 2. `tools/gen_status.zig` — pure parse → render, run as `zig build gen-status -- <args>`

Wired in `build.zig` next to the other `tools/*.zig` runners (e.g. the `conformance` runner at
`build.zig:161-178`). **Spawns nothing** — `just` owns orchestration; the tool only reads the three
captured output files, parses, and either writes or checks. This keeps it (a) unit-testable with
canned strings and (b) free of the `zig build` → `zig build` recursion footgun.

CLI:

```
gen-status --mode {write|check} \
           --test   .status-cache/test.txt \
           --hlsl   .status-cache/hlsl.txt \
           --conf   .status-cache/conformance.txt \
           --allowlist tests/known-conformance-fails.txt \
           --zig-version <v>
```

- `--mode write` → overwrite `docs/STATUS.md` + replace marker bodies in the 5 docs. Exit 0.
- `--mode check` → render the same content in memory, compare against on-disk; print what drifted;
  `exit(1)` if anything differs. No writes. (The standing drift guard.)

**Parse contracts** (stable runner / Zig output):

| Input | Parse | Yields |
|---|---|---|
| `test.txt` (`zig build test --summary all`) | `Build Summary: A/B steps succeeded; M/N tests passed`; hard-error if `M ≠ N` (no known unit fails) | `unit.tests = M`; best-effort per-`run test` lines |
| `hlsl.txt` (`zig build test-hlsl --summary all`) | same summary line | `hlsl.tests = M` |
| `conformance.txt` (`zig build conformance`) | per-fixture `PASS/FAIL …(spirv-val)/FAIL …(compile error)/SKIP <path>` lines grouped by `=== <suite> ===` headers, **plus** the `=== SUMMARY ===` block (`PASS:/FAIL (spirv):/FAIL (compile):/SKIP:/TOTAL:`, `tests/runner.zig:328-333`) | per-suite counts + the **set of failing fixture paths** + grand totals |

Cross-check: Σ(per-suite) **must** equal the SUMMARY block; mismatch → error (guards against a parse
or output-format change silently under-counting).

**Regression guard — the core anti-drift mechanism.** The observed set of failing fixtures is
diffed against the allowlist:

- observed-fails ⊆ allowlist → emit `conformance.fail = |observed|`, label "known feature-gap fails". ✅
- a fixture fails that is **not** in the allowlist → **abort loudly** (`error: regression — <path> is
  failing but not in tests/known-conformance-fails.txt`). Never publishes a clean number that hides a
  regression. This is what makes the old "2,087/2,087 zero-failures" lie structurally impossible.
- an allowlisted fixture now **passes** → warn (`<path> now passes — remove it from the allowlist`),
  do not fail.

### 3. `just update-status` (writer) + `just status` (read-only drift check)

`just` orchestrates; recipes run the three suites **sequentially** (the >2 GB-RAM-per-instance
constraint forbids parallel), tee combined stdout+stderr into a gitignored `.status-cache/`, then
call the tool. `conformance` is allowed to `exit(1)` (the 7 known fails) — the recipe captures its
output regardless (`zig build conformance | tee … ; true`-style) and lets the tool's regression
guard decide pass/fail.

- `just update-status` → run 3 suites → `gen-status --mode write` → docs updated.
- `just status` → run 3 suites → `gen-status --mode check` → prints the live table; `exit(1)` if the
  committed docs no longer match a fresh run. CI/pre-commit-ready.

Both pass `--zig-version $(mise exec -- zig version)` so the tool need not spawn.

### 4. `tests/known-conformance-fails.txt` — the allowlist

Checked-in, one fixture per line, `path<TAB># reason`. The single source of truth for "known fail vs
regression". Seeded with the 7:

```
tests/glslang-430/fp64.desktop.comp       # 64-bit float type not modelled (honest gap)
tests/glslang-430/int64.desktop.comp      # 64-bit int type not modelled (honest gap)
tests/glslang-430/newTexture.frag         # OpExtInst word-count on new-form texture builtins
tests/glslang-430/spv.newTexture.frag     # OpExtInst word-count on new-form texture builtins
tests/.../shader_ballot.comp              # subgroup ballot feature gap
tests/.../ray_sphere_test.frag            # feature gap
tests/.../struct-material.frag            # feature gap
```

(Exact paths resolved during implementation by reading the real `FAIL` lines from a fresh
conformance run — Task 1 of the plan. Lines starting `#` and blank lines ignored.)

## Marker scheme (the 5 prose docs)

Each restated number is wrapped **once, by hand** (one-time edit) in a comment-delimited region; the
generator rewrites only the inner text:

```html
<!-- STATUS:conformance.summary -->2,080 PASS / 7 FAIL / 8 SKIP / 2,095 TOTAL<!-- /STATUS -->
```

Injected scalar keys: `conformance.pass`, `conformance.fail`, `conformance.skip`,
`conformance.total`, `conformance.runnable`, `conformance.summary`, `unit.tests`, `hlsl.tests`.
Values are thousands-separated (`2,080`) to match existing prose. Rules:

- Every `<!-- STATUS:key -->` found must reference a **known key**; unknown key → error (catches typos).
- A key need not appear in every doc; docs use the subset they need.
- **Granular tables** (the stale per-suite §1.6 table in `IMPLEMENTATION_STATUS.md:113-128`, the
  per-stage `1,550` figures, the per-backend counts the file itself admits are stale at `:181`) are
  **not** marker-maintained — they are replaced by a link to the canonical table in `docs/STATUS.md`.
  Only headline scalars stay inline via markers. (Hybrid rule: scalars inline, tables centralized.)

## Testing (TDD)

`gen_status` parsing/rendering is pure and unit-tested against canned strings — **no suite run
needed**:

1. canned `test --summary all` sample → asserts `unit.tests`; a `M<N` sample → asserts hard-error.
2. canned `conformance` SUMMARY + per-fixture sample (incl. the 7 `FAIL` lines, multiple `=== suite ===`
   sections) → asserts per-suite counts, grand totals, Σ cross-check.
3. **regression sample**: an 8th `FAIL` not in the allowlist → asserts the tool errors.
4. **now-passing sample**: an allowlisted fixture absent from `FAIL` lines → asserts warn, not fail.
5. marker substitution over a small in-memory doc → asserts inner-text replacement; unknown-key doc
   → asserts error.

These live in `tools/gen_status.zig` `test {}` blocks (or a sibling test module wired as
`zig build test-gen-status`).

## Verification (task requirement: "numbers match a fresh local run")

By construction `update-status` renders from a fresh run, so the committed docs match reality the
moment they are written. The standing proof is `just status` (check mode) against a fresh run
exiting 0. The implementation will finish by running `just update-status` then `just status` and
pasting the SUMMARY block + a clean `git diff` of the regenerated docs as evidence.
